# Общие правила agentic-разработки для локального хранилища

## Суть
В глобальные правила хранилища нужно писать не проектную специфику, а устойчивый operating model для любой разработки с агентом. Лучший общий стандарт для вашего режима такой: короткие глобальные правила как policy-слой, repo-local docs как system of record, обязательный ask/plan перед кодом, работа маленькими milestones, жёсткие quality gates, явные `review queues` для неоднозначных решений и запрет silent fallback на критическом пути.

## Детали

## Что должно жить в глобальных правилах хранилища

### 1. Глобальные правила должны быть короткими и стабильными
В глобальные правила стоит класть только то, что не зависит от конкретного проекта:
- где лежат активные проекты, знания и логи;
- обязательный bootstrap нового треда;
- owner-only режим и матрицу автономии;
- обязательные quality gates;
- правила работы с индексами, логами и проектным контуром;
- общую политику по Git, секретам, destructive-операциям и релизам.

Не надо складывать в глобальные правила:
- детали архитектуры конкретного продукта;
- длинные research notes;
- специфику отдельного стека или одной команды;
- большие prompt-рыбы под один проект.

### 2. Для каждого кодового проекта должен быть свой system of record
Глобальные правила должны требовать, чтобы в любом новом проекте рядом с кодом жили:
- `AGENTS.md` как карта проекта;
- `ARCHITECTURE.md`;
- `docs/design-docs/`;
- `docs/data-contracts/`;
- `docs/exec-plans/active/`;
- process-артефакты `AGENT_WORKFLOW.md`, `TEST_PLAN.md`, `RELEASE_CHECKLIST.md`, `CHANGELOG.md`, `INCIDENT_RUNBOOK.md`.

Принцип простой: агент не должен вытаскивать критичный контекст из памяти чата. Источник правды должен лежать в проекте.

### 3. Перед кодом обязателен ask/plan phase
Общее правило для любой разработки:
- сначала агент читает локальные правила и repo-local docs;
- потом формирует self-contained execution plan;
- только после этого начинает код.

План должен фиксировать:
- цель и пользовательский результат;
- scope и non-goals;
- входные документы и исходники;
- milestones;
- acceptance criteria через наблюдаемое поведение;
- test strategy;
- риски и rollback path.

Если плана нет, агент почти всегда начинает латать локальные симптомы вместо проектного решения.

### 4. Задачи нужно давать маленькими, законченными milestones
Глобальная политика должна запрещать vague-задачи формата:
- "переделай систему";
- "сделай новую архитектуру";
- "почини всё".

Нормальный unit of work для агента:
- один milestone;
- примерно час инженерной работы или несколько сотен строк;
- понятный артефакт на выходе;
- локально проверяемый результат.

### 5. Глобальные правила должны разделять policy, memory и implementation docs
Хорошая трёхслойная модель:
- `global policy` — общие правила хранилища;
- `project memory` — устойчивые project-specific инструкции и known quirks;
- `execution plan` — что делаем в текущем цикле;
- `code/docs` — фактическая реализация.

Это уменьшает дрейф и не даёт агенту тащить случайный контекст в новый проект.

### 6. Любая agentic-разработка должна быть observability-first
Если агент или скрипт работает в цикле, одного `stdout` недостаточно.

Глобальное правило:
- agentic scripts обязаны писать структурированные логи;
- лог должен содержать `run_id`, `step_id`, `belief_state`, `observed_result`, `next_action`;
- должны быть `anti-loop protection` и forced context на повторном падении;
- без этого статус `GREEN` запрещён.

### 7. На критическом пути запрещены silent fallback и скрытая деградация
Это стоит записать как общий принцип, а не только для AI Digest.

Если critical step зависит от:
- модели;
- внешнего API;
- схемы structured output;
- обязательного поля данных,

то при неудаче система должна:
- либо явно упасть;
- либо создать review queue / degraded mode с явной маркировкой,

но не молча подменять логику "попроще".

### 8. Неоднозначные решения должны попадать в review queue
Во всех задачах, где агент:
- матчится по сущностям;
- мержит записи;
- нормализует справочники;
- выбирает из нескольких трактовок,

нужен общий паттерн:
- `confident` -> применить;
- `uncertain` -> отправить в review queue;
- `unsafe` -> остановиться.

Это особенно важно для графов, CRM, дедупликации, миграций и нормализации данных.

### 9. Review и implementation нужно разделять по контексту
Общее правило: не делать кодинг, отладку и финальный review в одном длинном контексте.

Нормальный паттерн:
- ask/plan;
- implementation;
- fresh-context review;
- test gate;
- release decision.

Иначе агент начинает защищать собственные предыдущие решения вместо независимой проверки.

### 10. Глобальные правила должны требовать evals, а не только "кажется ок"
Для любой сложной agentic-логики нужно требовать eval harness.

Минимум:
- golden examples;
- regression set;
- явные failure classes;
- метрики успеха для critical behavior.

Если логика опирается на LLM, без evals нельзя считать систему управляемой.

## Что лучше держать не в глобальных правилах, а в проекте

В проекте, а не в vault policy, нужно хранить:
- конкретную архитектуру продукта;
- data contracts;
- naming conventions и бизнес-термины;
- reuse boundaries для legacy;
- anti-patterns именно этого проекта;
- prompt templates для конкретного домена;
- eval datasets и scoring rubrics.

Правило: чем ближе знание к одному репозиторию, тем ближе к коду оно должно лежать.

## Базовый workflow, который стоит сделать обязательным для любой разработки

### Шаг 1. Bootstrap
Агент обязан:
- прочитать глобальные правила;
- проверить индексы и контур проекта;
- собрать только релевантный context pack;
- не сканировать весь vault без необходимости.

### Шаг 2. Ask-mode plan
Агент не пишет код, пока не соберёт:
- цель;
- ограничения;
- acceptance criteria;
- milestones;
- test plan;
- rollback idea.

### Шаг 3. Repo bootstrap
Если проект новый или poorly instrumented, агент сначала создаёт:
- process-артефакты;
- repo-local docs;
- execution plan;
- runbook.

### Шаг 4. Реализация коротким milestone
Один цикл = один milestone с понятным выходом.

### Шаг 5. Независимая проверка
Отдельный review pass в свежем контексте:
- correctness;
- regressions;
- edge cases;
- tests quality;
- observability.

### Шаг 6. Release decision packet
Пользователь получает только:
- `GREEN / YELLOW / RED`;
- что изменилось;
- что проверено;
- что нужно решить владельцу.

## Какой prompt-стандарт имеет смысл закрепить глобально

Для любых задач разработки достаточно одного общего шаблона постановки:

```text
Задача: <что изменить>
Цель: <какой пользовательский результат нужен>
Контекст: <какие файлы и документы считать source of truth>
Ограничения: <что нельзя ломать/трогать>
Scope: <что входит>
Non-goals: <что не входит>
Артефакты на выходе: <какие файлы/доки/тесты должны появиться>
Acceptance criteria: <какое наблюдаемое поведение считается успехом>
Порядок работы: сначала ask/plan, потом реализация milestone-by-milestone
```

Этого достаточно, если в глобальных правилах уже закреплены:
- owner-only режим;
- обязательные тесты;
- release gate;
- обновление индексов и проектного контура.

## Антипаттерны, которые стоит прямо запретить
- Начинать код без ask/plan.
- Пытаться чинить legacy по месту, если нужен greenfield-redesign.
- Давать агенту сразу слишком широкий scope.
- Смешивать global policy и проектную специфику в одном файле.
- Оставлять critical decisions без evals.
- Молчаливо деградировать в heuristic/manual fallback.
- Завершать задачу без тестов, smoke и rollback-пути.
- Пытаться держать весь контекст только в чате.

## Что я бы рекомендовал добавить в ваши локальные правила как постоянный стандарт
1. Для любой разработки сначала `ask/plan`, потом код.
2. Глобальные правила короткие; проектная специфика живёт рядом с кодом.
3. Каждый кодовый проект обязан иметь repo-local system of record и process-артефакты.
4. Любая LLM-критичная логика требует evals и запрета silent fallback.
5. Agentic scripts без structured logs и anti-loop protection не могут получать `GREEN`.
6. Неоднозначные автоматические решения обязаны уходить в review queue.
7. Implementation и review должны идти разными фазами и по возможности в разном контексте.
8. Новый крупный продукт по умолчанию запускать как отдельный track, а не как бесконечный refactor legacy.
