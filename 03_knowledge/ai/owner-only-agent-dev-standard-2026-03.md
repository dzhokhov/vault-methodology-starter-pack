# Owner-only стандарт разработки с LLM-агентом

## Суть
Стандарт для пользователя без технической экспертизы: агент делает реализацию и полную самопроверку, пользователь принимает только минимальные управленческие решения (цель и выпуск/откат).

Связанный индекс раздела: [Индекс знаний](../README.md).

## Детали

### 1) Обязательный scope
Применяется ко всем задачам, где агент пишет или меняет:
- скрипты (`scripts/` и соседние репозитории),
- программы/приложения (например Swift/AppKit, Python, CLI),
- CI/CD и release-конфигурации.

### 2) Роли и ответственность
- Пользователь (Owner):
  - задаёт цель и приоритет;
  - принимает решение `Release` или `Rollback` для релизных изменений;
  - не участвует в технической реализации.
- Агент:
  - выполняет декомпозицию, реализацию, тестирование и сборку;
  - проводит обязательную самопроверку перед любым статусом “готово”;
  - готовит release packet с оценкой риска и планом отката.

### 3) Обязательные артефакты процесса в кодовом проекте
Перед существенными изменениями в проекте должны быть:
- `AGENT_WORKFLOW.md`
- `TEST_PLAN.md`
- `RELEASE_CHECKLIST.md`
- `CHANGELOG.md`
- `INCIDENT_RUNBOOK.md`

Если артефактов нет — агент обязан создать их из шаблонов:
- [Шаблон AGENT_WORKFLOW](.../meta/templates/agent_workflow.md)
- [Шаблон TEST_PLAN](.../meta/templates/test_plan.md)
- [Шаблон RELEASE_CHECKLIST](.../meta/templates/release_checklist.md)
- [Шаблон CHANGELOG](.../meta/templates/changelog.md)
- [Шаблон INCIDENT_RUNBOOK](.../meta/templates/incident_runbook.md)
- [Шаблон owner decision packet](.../meta/templates/owner_decision_packet.md)

### 4) Автономия агента: что можно и что запрещено
- Агент может автономно:
  - писать код в рамках scope;
  - запускать lint/type/static checks;
  - запускать unit/integration/smoke;
  - обновлять changelog и checklist.
- Агенту запрещено без явного подтверждения пользователя:
  - destructive операции (`rm -rf`, history rewrite, force push);
  - изменение секретов/доступов/permissions;
  - production deploy high-risk изменений;
  - миграции данных без подтверждённого rollback-пути.

### 5) Контур самопроверки (обязательный)
Перед статусом “готово” агент обязан пройти:
1. `Code gate`: lint/type/static/security checks.
2. `Test gate`: unit + integration по затронутому scope.
3. `Release gate`: сборка артефакта + smoke на чистом окружении.
4. `Rollback gate`: подтверждённый путь отката (tag/revert/feature-flag).
5. `Adversarial self-check`: что могло сломаться и как это обнаружится.

Если хотя бы один gate не пройден — `GREEN` запрещён.

### 5.1) Дополнительный gate для agentic development

Если изменение касается:
- автономных агентов,
- скриптов, которые агент запускает в цикле,
- self-healing / self-debugging сценариев,
- test-fix-retry контуров,

то обязателен **observability gate**.

Минимум:
- есть `agent-readable` runtime-логи, а не только человеко-читаемый вывод;
- лог содержит `log-to-code correlation`;
- лог фиксирует `belief_state` агента, а не только итоговый error text;
- есть `anti-loop protection`;
- на повторном падении существует `forced context` или другой механизм подачи памяти о прошлых неудачах.

Если агент работает в цикле и этих механизмов нет, статус `GREEN` запрещён даже при проходящих тестах.

### 6) Формат коммуникации с владельцем (только светофор)
Агент возвращает только один из статусов:
- `GREEN`: всё проверено, риск низкий/контролируемый, можно выпускать.
- `YELLOW`: есть ограничение, нужен выбор владельца из 1-2 вариантов.
- `RED`: выпуск запрещён, требуется откат/исправление.

Для каждого статуса агент добавляет:
- что изменилось (1-2 строки человеческим языком),
- что проверено (короткий список),
- какой шаг нужен от владельца (`Release`, `Hold`, `Rollback`).

### 7) Минимальный вход от владельца
Владелец даёт задачу форматом:
```text
Задача: <что изменить>
Цель: <какой результат для пользователя>
Дедлайн: <когда нужно>
Риск: low | medium | high (если не уверен — medium)
Ограничения: <что нельзя трогать>
```
