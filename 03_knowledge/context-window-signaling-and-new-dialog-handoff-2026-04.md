# Сигнал остатка контекста и переход в новый диалог

## Суть
Для глобального правила лучше использовать не точный token count, а очень короткий эвристический хвост `CTX~N%`. При критическом остатке агент не должен «дожимать» длинный ответ в умирающем чате: безопаснее предложить `new-dialog-handoff`, опереться на `parking`/`resume` и продолжить в свежем контексте из durable source of truth.

## Детали

### TL;DR
- Лучший формат хвоста: одна строка, без пояснений, с грубой оценкой в процентах.
- Точный остаток окна в чат-интерфейсе обычно недоступен; даже API token counting у Anthropic описан как estimate, не как абсолютная истина.
- Порог лучше делать не по «жёсткому числу токенов», а по прогнозу следующего ответа: если следующий содержательный шаг, вероятно, не влезет, агент предлагает handoff.
- Handoff не должен быть свободным пересказом чата. Источник правды должен лежать в файлах: `log.md`, `tasks.md`, `context.md`, research-note, spec.
- Для проектной работы нужно переиспользовать `parking`, а в следующем чате входить через `resume`.
- Длинный контрольный review лучше делать в свежем контексте, а не в том же разросшемся чате.

### Что показывать в конце каждого сообщения
Рекомендуемый глобальный хвост:

```text
CTX~65%
```

Критический вариант:

```text
CTX~10% -> new-dialog-handoff
```

Почему так:
- коротко;
- стабильно;
- почти не расходует токены;
- понятно человеку без расшифровки;
- не делает вид, что агент знает точный token balance.

### Почему не стоит требовать точный счётчик
Официальные API умеют считать токены и управлять compaction/state, но это не означает, что агент в чат-продукте видит точный остаток окна на каждом ходу. У Anthropic token counting прямо описан как estimate. Поэтому в глобальном правиле безопаснее закрепить:
- грубую оценку;
- округление до 5-10%;
- явный прогнозный триггер на handoff;
- запрет выдавать число как точный остаток.

### Когда предлагать новый диалог
Рабочий порог:
- `CTX~15%` и ниже;
- или агент видит, что следующий ответ будет длинным: код, review, ресёрч, многофайловая правка, длинное объяснение.

Решение:
- не продолжать длинное развёртывание;
- предложить `new-dialog-handoff`;
- сначала зафиксировать состояние в durable source of truth;
- только потом переносить работу в свежий чат.

Это лучше, чем пытаться «втиснуть ещё один большой ответ» в почти исчерпанное окно: качество ответа падает, а риск потери нитки разговора растёт.

### Какой handoff считается хорошим
Хороший handoff:
- опирается на уже существующие артефакты;
- не дублирует `parking` и `resume`;
- даёт следующий стартовый prompt максимум в 4 пунктах;
- указывает 1-3 canonical file path как source of truth;
- фиксирует ровно один следующий шаг.

Плохой handoff:
- длинный пересказ всей сессии;
- новая параллельная память вместо `log.md`/`tasks.md`/research-note;
- попытка держать критичный контекст только в чате.

### Рекомендуемый operating model для этого vault
1. В `AGENTS.md` закрепить обязательный хвост `CTX~N%`.
2. При критическом прогнозе предлагать `new-dialog-handoff`.
3. `new-dialog-handoff` должен:
   - для проектной работы сначала вызывать протокол `parking`;
   - для следующего чата опираться на `resume`;
   - при наличии готового durable artifact не создавать параллельную память;
   - вернуть короткий restart-packet, а не длинный recap.

### Рекомендуемый текст глобального правила
```text
В конце каждого сообщения добавляй одну строку: `CTX~N%`.
`N` — грубая эвристическая оценка остатка окна после текущего ответа, округлённая до 5-10%.
Если по прогнозу следующий содержательный ответ может упереться в лимит, вместо обычного хвоста пиши:
`CTX~N% -> new-dialog-handoff`
и предлагай перейти в свежий диалог через этот скилл. Не выдавай `N` как точный token count.
```

### Почему это согласуется с локальными правилами
- [AGENTS.md](../../AGENTS.md) требует держать глобальные правила короткими и стабильными.
- [parking](../../skills/parking/SKILL.md) уже фиксирует точку возврата и прямо рекомендует новый чат, если разговор стал длинным.
- [resume](../../skills/resume/SKILL.md) уже задаёт путь обратного входа в припаркованную задачу.
- [Общие правила agentic-разработки для локального хранилища](./general-rules-for-agentic-development-in-local-vault-2026-03.md) прямо говорят не держать весь контекст только в чате и разделять implementation/review по свежему контексту.

### Blind spots и ограничения
- Порог `15%` эвристический, не математический.
- В разных продуктах точное поведение окна отличается; локальное правило должно быть продукт-agnostic.
- Если у платформы позже появится настоящий visible token meter, правило стоит упростить и перевести с эвристики на реальный счётчик.

## Следующий шаг
Если правило покажет себя стабильно в нескольких длинных сессиях, можно оставить только короткий хвост в `AGENTS.md`, а детальные обоснования держать в этой заметке.
