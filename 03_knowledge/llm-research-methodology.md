# Методология исследований с помощью LLM

## Суть
Систематизированные best practices проведения исследований с помощью больших языковых моделей: как получить качественный, релевантный результат, какие искажения ожидать и как их компенсировать.

## Детали

### Фундаментальные ограничения LLM как исследовательского инструмента

**1. Галлюцинации (hallucinations)**
28% выходных данных LLM содержат правдоподобную, но фактически неверную информацию. Различают: entity hallucinations (ошибки в именах, компаниях, датах, версиях API), attribution hallucinations (факт верный, но приписан не тому источнику), intrinsic (внутренняя непоследовательность ответа) и extrinsic (противоречие внешним фактам).

**2. Нестабильность результатов**
Результаты варьируются в 36.4% повторных запросов. Один и тот же вопрос может дать разные ответы в разных сессиях. Это не баг — это свойство стохастической генерации.

**3. Training data bias**
LLM воспроизводят и усиливают искажения из обучающих данных. Для коммерческих тем это особенно критично: в training data непропорционально много контент-маркетинга, SEO-статей и рекламных материалов. Модель «думает» так, как «думает» интернет — а интернет непропорционально представляет голоса тех, кто продаёт.

**4. Позиционное смещение (position bias)**
При оценке нескольких вариантов LLM систематически отдаёт предпочтение первому или последнему варианту. GPT-4 показывает до 40% непоследовательности в зависимости от порядка подачи.

**5. Verbosity bias**
LLM считает более длинные, подробные ответы более качественными. Более многословный ответ получает ~15% завышения оценки.

**6. Self-enhancement bias**
Модели предпочитают контент, сгенерированный другими LLM, с завышением на 5-7%. AI предпочитает AI — это подтверждено исследованием PNAS 2025.

### Карта искажений при LLM-ресёрче

#### Искажение «крикунов» (Vocal Minority Bias)

**Проблема:** при исследовании любой коммерческой темы LLM (и поисковые системы) возвращают преимущественно мнения трёх групп:
- **Продавцы** — компании, продвигающие свой продукт через контент-маркетинг. Их статьи SEO-оптимизированы и занимают топ выдачи.
- **Крикуны** (vocal minority) — люди с сильными эмоциями (негативными или позитивными), которые пишут отзывы и посты. Молчаливое большинство не представлено.
- **Евангелисты** — энтузиасты, которые продвигают определённые подходы/инструменты из убеждений.

Исследования MIT Sloan и ScienceDaily (2025) подтверждают: малая группа активных пользователей создаёт большую часть контента, люди переоценивают степень токсичности и поляризации интернета, мнения в social media систематически смещены vocal minority.

**Компенсация:**
- Явно запрашивать альтернативные точки зрения: «Какие аргументы ПРОТИВ этого подхода?»
- Запрашивать источники из академических/независимых исследований, а не блогов
- Отдельно спрашивать: «Кто из отвечающих на этот вопрос имеет коммерческий интерес?»
- Запрашивать «молчаливое большинство»: «Что делает типичный пользователь, который не пишет посты?»

#### Survivorship Bias

**Проблема:** LLM знает про успешные компании, продукты, подходы — но не знает про тех, кто пробовал то же самое и провалился. В training data мало постмортемов и много success stories.

**Компенсация:**
- Явно спрашивать: «Какие компании/проекты НЕУСПЕШНО пробовали этот подход? Почему?»
- «Каковы типичные причины провала при использовании [X]?»
- Отдельно искать постмортемы и failure reports

#### Authority Bias

**Проблема:** LLM переоценивает мнения крупных компаний (FAANG, McKinsey) и недооценивает практики малого бизнеса, нишевых игроков, развивающихся рынков.

**Компенсация:**
- «Как эта проблема решается в компаниях до 50 человек / на рынках СНГ / в стартапах?»
- «Есть ли противоположное мнение от практиков, а не консультантов?»

#### Recency Bias / Outdated Knowledge

**Проблема:** LLM может выдавать устаревшую информацию, особенно для быстро меняющихся областей (SaaS-инструменты, API, регуляторика). Или, наоборот, переоценивать новейшие тренды.

**Компенсация:**
- Всегда уточнять: «Это актуально на [текущая дата]?»
- Использовать web search для верификации
- Спрашивать про эволюцию: «Как этот подход менялся за последние 3 года?»

#### Контент-маркетинговый шум (Commercial Content Noise)

**Проблема:** по коммерческим запросам LLM неизбежно включает знания из контент-маркетинговых статей, которые продвигают конкретные продукты/подходы.

**Компенсация:**
- Запрашивать принципы, а не инструменты: «Какие ПРИНЦИПЫ лежат в основе?» вместо «Какой инструмент лучший?»
- Просить классификацию: «Раздели ответ на: (1) факты, (2) мнения, (3) рекламные утверждения»
- Просить disclaimer: «Отметь, если какое-то утверждение может исходить из коммерческого источника»
- Использовать negative prompting: «Не рекомендуй конкретные коммерческие продукты, опиши критерии выбора»

### Практические техники качественного LLM-ресёрча

#### Техника 1: Триангуляция (Triangulation)

Концепция из научной методологии: использовать минимум 2-3 независимых метода для исследования одного вопроса. Для LLM-ресёрча:

- **Между моделями:** задать один вопрос нескольким LLM (ChatGPT, Claude, Gemini). Если ответы расходятся — это зона неопределённости. Исследования показывают, что использование 3-5 моделей с majority vote снижает искажения на 30-40%.
- **Между модальностями:** LLM-ответ + web search + академические источники.
- **Между ролями:** попросить ту же LLM ответить от лица разных экспертов: «Ответь как практик-фрилансер», «Ответь как исследователь», «Ответь как скептик».

#### Техника 2: Chain of Verification (CoV)

Принцип: после генерации ответа LLM проверяет каждое утверждение. Практическая реализация:

1. Получи основной ответ
2. Попроси: «Перечисли все фактические утверждения в этом ответе»
3. Для каждого утверждения: «Насколько ты уверен? Какой источник?»
4. Отдельно проверь через web search самые критичные утверждения

#### Техника 3: Adversarial Prompting (проверка на прочность)

После получения ответа:
- «Какой самый сильный аргумент ПРОТИВ этого вывода?»
- «Представь, что ты критик — какие дыры в этом анализе?»
- «Что я упускаю? Какие blind spots?»
- «Если бы я показал этот вывод [скептически настроенному эксперту], что бы он сказал?»

#### Техника 4: Decomposition (декомпозиция сложных запросов)

Разбить большой исследовательский вопрос на 5-7 подвопросов и исследовать каждый отдельно. Это снижает галлюцинации, потому что:
- Модель фокусируется на узком контексте
- Легче верифицировать каждый подответ
- Меньше соблазн для модели «заполнить пробелы» выдумкой

#### Техника 5: Source-First Research (от источников, а не от выводов)

Вместо «Расскажи про X» → «Найди 5 первоисточников (не блоги) по теме X, кратко опиши каждый и приведи ключевые цифры/выводы».

Это заставляет модель работать от фактов, а не от нарратива.

#### Техника 6: Явная маркировка уверенности

В промпте: «Для каждого утверждения укажи степень уверенности: ✅ подтверждено множеством источников / ⚠️ есть в нескольких источниках, но спорно / ❓ моё предположение на основе паттернов / 🔴 не могу подтвердить».

Это принуждает модель к рефлексии вместо уверенной генерации.

#### Техника 7: Red Team промпт

Отдельный шаг после ресёрча: «Ты — red team. Твоя задача: найти все ошибки, неточности, необоснованные утверждения и коммерческие искажения в этом тексте. Будь безжалостен.»

### Структура идеального ресёрч-промпта

```
Роль: Ты — [конкретная экспертная роль], исследующий [тему].

Контекст: Мне нужно понять [что именно] для [какой цели].
Моя ситуация: [специфика — размер компании, рынок, ограничения].

Задача: Исследуй [конкретный вопрос].

Ограничения:
- Приоритет: академические/независимые источники > практический опыт > блоги/контент-маркетинг
- Отмечай степень уверенности для каждого утверждения
- Разделяй факты, мнения и рекламные утверждения
- Явно указывай, если информация может быть устаревшей
- Не рекомендуй конкретные коммерческие продукты, опиши критерии

Формат вывода:
1. TL;DR (3-5 предложений)
2. Основные находки (с маркировкой уверенности)
3. Альтернативные точки зрения / контраргументы
4. Blind spots и ограничения этого исследования
5. Источники (с пояснением типа: академический / практический / коммерческий)
6. Что нужно проверить дополнительно
```

### Workflow LLM-ресёрча (пошаговый)

1. **Декомпозиция.** Разбей вопрос на подвопросы. Каждый подвопрос — отдельный запрос.
2. **Первый проход.** Получи базовые ответы. Не верь им — это гипотезы.
3. **Триангуляция.** Проверь ключевые утверждения через web search и/или вторую LLM.
4. **Adversarial check.** Попроси LLM атаковать собственные выводы.
5. **Source verification.** Проверь ключевые источники — существуют ли они? Верно ли цитируются?
6. **Синтез.** Объедини результаты, отмечая степень уверенности.
7. **Red team.** Финальная проверка на ошибки и искажения.
8. **Сохранение.** Запиши результат в базу знаний со ссылками на источники.

### Искажения исследователя (человеческий фактор)

Документ выше описывает bias модели. Но у исследователя есть собственные когнитивные искажения, которые усиливают проблему.

**Confirmation bias (предвзятость подтверждения)**
Исследователь формулирует вопрос так, что уже подсказывает «правильный» ответ. «Почему Obsidian лучше Notion?» вместо «Сравни Obsidian и Notion для моих задач». LLM охотно подстраивается под направление вопроса — sycophancy bias модели + confirmation bias человека = замкнутый круг.

Компенсация: формулировать вопросы нейтрально, начинать с открытых вопросов, осознанно искать опровержения своей гипотезы.

**Anchoring (эффект якоря)**
Первый полученный ответ становится якорем. Все последующие ответы оцениваешь относительно него. Если первая модель сказала «X — лучший подход», сложно объективно оценить альтернативы.

Компенсация: начинать триангуляцию с разных моделей одновременно, менять порядок подвопросов между сессиями, явно фиксировать: «Мой текущий якорь — X. Какие аргументы за Y?»

**Framing effect (эффект фрейминга)**
Как поставил вопрос — так получил ответ. «Какие риски у подхода X?» и «Какие преимущества подхода X?» дают совершенно разные ответы от одной модели.

Компенсация: задавать один вопрос в нескольких фреймах, явно просить: «Опиши и плюсы, и минусы, и нейтральную позицию».

**Over-researching (исследовательский паралич)**
Бесконечное углубление в детали вместо перехода к действию. Признаки: новые запросы возвращают уже известное, контраргументы повторяются, уровень уверенности по ключевым вопросам стабилизировался. Если три разных источника говорят одно и то же — ты достиг насыщения.

Компенсация: до начала ресёрча определить: (1) какое решение я принимаю, (2) какой допустимый уровень неопределённости, (3) дедлайн на ресёрч. По достижении любого из трёх — остановиться.

### Языковое и культурное смещение

Training data LLM на 60-70% состоит из англоязычного контента. Это приводит к системному смещению:

- **Географический bias**: модель по умолчанию мыслит паттернами US/EU рынка. Для российского, азиатского, латиноамериканского контекста best practices могут не работать.
- **Культурный bias**: стиль коммуникации, бизнес-нормы, правовые рамки — всё завязано на англосаксонскую традицию.
- **Языковой bias**: на русском языке модель имеет доступ к меньшему объёму данных и может давать менее точные ответы, особенно по нишевым темам.

**Компенсация:**
- Явно указывать географию и контекст: «для российского рынка / для компании в СНГ / для русскоязычной аудитории»
- Проводить часть запросов на английском — это открывает доступ к другому пласту данных
- Перепроверять: «Этот паттерн работает на моём рынке, или это западная специфика?»
- Для регуляторных и правовых вопросов — всегда верифицировать через локальные источники

### Разница между «узнать» и «решить»

Исследование ради знания и исследование ради решения — разные процессы с разными критериями качества.

**Исследование-знание:** «Как устроены Personal CRM?» → широкий обзор, множество точек зрения, максимальный охват.

**Исследование-решение:** «Какую Personal CRM мне выбрать?» → узкий фокус на моих ограничениях, чёткие критерии, быстрый выход на действие.

Типичная ошибка: начать с исследования-решения, но скатиться в исследование-знание. Накапливать информацию приятно, принимать решение в условиях неопределённости — нет.

**Компенсация:**
- До начала определить тип: я хочу узнать или решить?
- Для решения — сформулировать критерии ДО ресёрча, а не после
- Зафиксировать ограничения (бюджет, время, навыки, экосистема) — они отсекают 80% вариантов
- Установить дедлайн: «Через 2 часа ресёрча я принимаю решение с тем, что есть»

### Мультисессионный протокол

Серьёзное исследование редко помещается в одну сессию LLM. Нужен протокол передачи контекста.

**Проблема:** каждая новая сессия начинается с нуля. Если просто описать результат словами — теряются нюансы, оговорки, степени уверенности.

**Протокол:**
1. В конце сессии — экстрагировать результаты в структурированный формат (промпт экстракции, см. примеры в проектах CJM и Personal CRM)
2. Сохранить в базу знаний со ссылками на источники
3. В начале новой сессии — дать модели прочитать сохранённый документ как контекст
4. Явно указать: «Это результаты предыдущего исследования. Не повторяй их, продолжай с этой точки.»
5. Периодически делать синтез: объединять результаты из нескольких сессий в один документ

**Антипаттерн:** пересказывать результаты прошлой сессии своими словами. Это вносит собственный bias пересказчика и теряет детали.

### Чеклист качества LLM-ресёрча

Перед финализацией результатов проверь:

- [ ] Есть ссылки на первоисточники (не только блоги)?
- [ ] Представлены альтернативные точки зрения?
- [ ] Отмечены утверждения с низкой уверенностью?
- [ ] Проверены ключевые факты через web search?
- [ ] Выявлены потенциальные commercial biases?
- [ ] Учтена vocal minority vs silent majority?
- [ ] Проверено на survivorship bias (есть примеры неудач)?
- [ ] Результат релевантен МОЕЙ ситуации (а не generic best practice)?
- [ ] Информация актуальна на текущую дату?
- [ ] Указаны ограничения и blind spots исследования?
- [ ] Проверен ли собственный confirmation bias (нейтральная формулировка вопросов)?
- [ ] Учтено языковое/культурное смещение (релевантно моему рынку)?
- [ ] Определён тип исследования (знание vs решение) и критерии остановки?
- [ ] Результаты сохранены в базу знаний для мультисессионного использования?
- [ ] Не скатился ли ресёрч в over-researching (насыщение данных)?
