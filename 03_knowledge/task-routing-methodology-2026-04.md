# Модель маршрутизации задач и контекста встреч

## Суть

Документ фиксирует целевую модель управления задачами, планом проекта, делегированиями, личными обязательствами и эпизодическим контекстом встреч. Модель пришла на смену прежнему подходу, где `tasks.md` проекта принимал все четыре класса сущностей сразу и превращался в свалку.

Этот файл — **источник истины методологии**. Правила в `meta/rules/` и скилл `meeting-processing` должны ссылаться сюда и не дублировать рассуждение.

---

## 1. Проблема, которую закрываем

Прежняя модель хранила в одном `tasks.md` проекта четыре разнородных сущности:
1. Execution-шаги агента по текущему milestone;
2. Milestones и план достижения цели;
3. Делегирования сотрудникам;
4. Личные внешние обязательства владельца, упомянутые на встрече.

Следствия:
- Срез «мои задачи» и «задачи делегата X» невозможен без обхода всех проектов.
- План проекта деградирует: стратегический milestone соседствует с микрозадачей, вектор теряется.
- `context.md` проекта засоряется эпизодическими фактами со встреч — и перестаёт быть устойчивым инвариантом.
- Скилл `meeting-processing` имеет одномерную модель «раскидай по tasks.md» и не различает владельца/слой/срок жизни.

## 2. Основной принцип

> Каждый класс задач живёт там, где его владелец и срок жизни, а не там, где он был впервые упомянут.

Отсюда все частные правила.

## 3. Пять классов сущностей и их места

| Класс | Описание | Место хранения |
|---|---|---|
| **Execution step агента** | Что агент делает в текущей сессии в рамках проекта | `01_now/projects/<project>/tasks.md` |
| **Milestone проекта** | Устойчивый шаг плана с acceptance criteria | `01_now/projects/<project>/plan.md` |
| **Делегирование** | Задача, где исполнитель не владелец | `01_now/ops/<contour>/delegations/<person-slug>.md` |
| **Личное внешнее обязательство** | То, что владелец обязан сделать лично, без проектной привязки | `01_now/personal/tasks.md` |
| **Эпизодический факт со встречи** | Упомянуто вскользь, может всплыть позже | `<project>/materials/<contour>/meetings/YYYY-MM-DD_тема.md`, секция `## Упомянуто вскользь`, под хэштегом темы |

Любой устойчивый инвариант (термин, метрика, source of truth) — это отдельный класс, он идёт в `context.md` проекта и в этой модели не участвует.

## 4. Decision tree агента при маршрутизации строки со встречи

Агент перед записью каждой строки задаёт **один** вопрос: «переживёт ли эта строка этот проект?».

```
Переживёт ли строка этот проект?
├── Нет
│   ├── Это шаг агента на текущем milestone? → <project>/tasks.md (Active/Next)
│   └── Это эпизодический факт? → meetings/YYYY-MM-DD.md, ## Упомянуто вскользь, #тег
└── Да
    ├── Исполнитель — владелец vault'а?
    │   ├── Часть цели проекта? → <project>/plan.md как milestone
    │   └── Внешнее обязательство (не проект)? → 01_now/personal/tasks.md, Hard
    ├── Исполнитель — сотрудник контура? → 01_now/ops/<contour>/delegations/<person>.md
    └── Это устойчивый инвариант (термин, метрика, SoT)? → <project>/context.md
```

Если ответ неоднозначный — агент спрашивает владельца, не угадывает (Правило 6).

## 5. Структура файлов проекта (operational режим)

```
01_now/projects/<project>/
├── README.md    — навигация
├── context.md   — УСТОЙЧИВЫЕ инварианты: термины, метрики, source of truth
├── plan.md      — ЖЁСТКИЙ скелет: Goal, Non-goals, Appetite, Milestones, Blockers, Blockers — Resolved, Drift Guard, Contingency
├── tasks.md     — КОРОТКИЙ: Active step, Exit criteria, Next. Только про исполнение, не про размышление.
├── log.md       — хронология событий (Правило 5)
└── materials/<contour>/meetings/YYYY-MM-DD_тема.md — саммари встреч, эпизодика под хэштегами
```

### 5.1. `plan.md` — контракт проекта

```markdown
# Plan: <проект>

## Goal
<одно предложение, наблюдаемый результат>

## Non-goals
<что в этом проекте НЕ делаем — защита от scope creep>

## Appetite
<сколько внимания/времени готов вкладывать, за какой срок переосмысление>
Пример: «час в день 2 недели, после — решение продолжать или закрыть».

## Source of truth
- Контекст: ./context.md
- Хронология: ./log.md
- Текущий шаг: ./tasks.md

## Milestones

### M1. <название>
- Что: <описание>
- Acceptance: <наблюдаемый критерий завершения>
- Статус: not-started | in-progress | done | blocked on Bx

### M2. ...

## Blockers

Каждый активный блокер — отдельная запись, на которую ссылаются milestone'ы через `blocked on Bx`. Пустая секция — нормальное состояние.

### B1 — <короткое имя блокера>
- Открыт: YYYY-MM-DD
- Тип: cross-project dependency | delegated-work | external-event
- Ждём: <что именно должно случиться, чтобы блок снялся>
- Блокирует: M2, M3 (список milestone'ов, каскад)
- Моё действие: passive | active-nudge | switch-to-fallback-at-YYYY-MM-DD
- Условие возврата: <как я пойму, что блокер снят>
- Fallback: <что делаем, если блокер не снимется к сроку>
- Stale check: <через сколько дней считать блокер устаревшим и поднимать вопрос>

## Blockers — Resolved

Журнал снятых блокеров. Живёт до ревью или forced rewrite, потом схлопывается. Не хроника log.md, а короткая сводка «что мешало и как разошлось», чтобы не забыть паттерн.

### B0 — <имя> (resolved YYYY-MM-DD)
- Было: <однострочное описание>
- Разошлось через: <что сделало блок нерелевантным>

## Drift Guard
Если агент хочет изменить подход, scope или порядок milestones:
1. Обновить этот файл (что меняется и почему).
2. Записать причину в log.md (одна строка).
3. Только потом начинать работу по новому плану.

## Contingency
<план Б на уровне всего проекта: что делаем, если основной путь заблокирован>
```

Заимствования из Shape Up (pitch): `Appetite` = appetite, `Non-goals` = no-gos, `Drift Guard` = rabbit holes, `Milestones` + `Goal` = solution. `Contingency` — одна секция на весь проект, не под каждым milestone (валидация 2026-04-15).

`plan.md` — медленный файл. Меняется, когда меняется цель, appetite или milestone.

### 5.2. `tasks.md` — короткий execution queue

```markdown
# Tasks: <проект>

## Режим
operational | development

## Active step
<одно предложение: что агент делает прямо сейчас>

## Exit criteria
- <наблюдаемое условие завершения>

## Next
- <1-3 пункта, что сразу после active>
```

**Правило содержимого:** `tasks.md` содержит только строки про **исполнение**, не про размышление. Если строка отвечает на вопрос «что непонятно?», «куда двигаться?», «какой путь выбрать?» — её место в `plan.md` (как milestone или как открытый вопрос под milestone). Если строка отвечает на «что сделать сейчас?» — в `tasks.md`. Секции `## Blocked` в `tasks.md` нет — блокеры живут в `plan.md §Blockers` как first-class сущность (см. §5.4).

Раздутый `tasks.md` — всегда симптом того, что туда попало размышление. Это проще диагностировать, чем считать строки.

### 5.3. Режим `development`

Для кодовых проектов `tasks.md` ведётся в режиме `development` (`Current Milestone / Active Step / Exit Criteria / Drift Guard`), `plan.md` всё равно обязателен и содержит стратегический скелет. Правило 10 из AGENTS.md остаётся в силе, но теперь plan/tasks физически разведены.

### 5.4. Блокеры как first-class сущность плана

Блокер — это не строка в `tasks.md` и не запись в `log.md`. Это **первоклассная сущность плана**, живущая в `plan.md` секции `## Blockers`, на которую ссылаются milestone'ы через поле статуса `blocked on Bx`. До 2026-04 блокеры жили в `tasks.md` как секция `## Blocked` — это оказалось ошибкой: они теряли связь с milestone, который блокируют, и не переживали rotation быстрого файла.

**Почему отдельная секция, а не Contingency.** Contingency — это план Б на уровне всего проекта («если appetite исчерпан — закрываем»). Блокер — это конкретный затор на конкретном шаге, с именем, сроком, ожидаемым событием и каскадом влияния на другие milestone'ы. Это разные масштабы, они не должны мешаться.

**Почему не log.md.** `log.md` — хронология. Он отвечает на вопрос «что произошло». `plan.md §Blockers` отвечает на вопрос «что сейчас мешает и где это в карте проекта». Запись о появлении блокера идёт и туда, и туда, но источник истины состояния — `plan.md`.

**Двусторонние ссылки.** Каждый блокер знает, что он блокирует (поле `Блокирует: M2, M3`). Каждый milestone знает, что он заблокирован (статус `blocked on Bx`). Это делает каскады явными: если B1 блокирует M2, а M3 зависит от выходов M2, — M3 тоже получает `blocked on B1`, и это видно в plan.md сразу.

**Cross-project dependencies — только one-way.** Если проект A ждёт результата из проекта B, блокер записывается только в `plan.md` проекта A (waiter). Проект B (parent) не знает о waiter'ах — иначе parent-план начинает раздуваться служебной информацией. Агент обнаруживает cascade при касании parent-проекта через grep: `rg "B-контур.*cross-project" 01_now/projects/*/plan.md`.

#### 5.4.1. Типы блокеров

| Тип | Когда | Источник действия |
|---|---|---|
| **cross-project dependency** | Жду результата из другого проекта vault'а | Passive — поднимется при касании parent-проекта |
| **delegated-work** | Делегирование на сотруднике, результат нужен для milestone | Параллельно — запись в `delegations/<slug>.md`, в `plan.md §Blockers` — только если снятие блока критично для milestone |
| **external-event** | Жду внешнего события (ответ клиента, дата, ответ госоргана) | Active nudge по `Моё действие` или passive до даты |
Тип фиксируется в карточке блокера, влияет на ritual нudge'а и на то, куда ещё запишется строка.

`need-info` **не является типом блокера**. Если у владельца есть действие, которое приближает результат, это не wait-state:
- тактический `need-info` → `tasks.md`;
- стратегический `need-info` → правка `plan.md` (`Milestones` / `Drift Guard`), но не карточка `Bx`.

#### 5.4.2. Ритуал разрешения блокера

Порядок шагов жёсткий (медленные слои опережают быстрые — [write-protocol.md §5](../meta/rules/write-protocol.md)):

1. Переместить карточку из `## Blockers` в `## Blockers — Resolved` (в plan.md), сократить до двух строк: «Было / Разошлось через».
2. Обновить статус каждого milestone из поля `Блокирует`: `blocked on Bx` → `in-progress` или `not-started` (что верно по фактическому состоянию).
3. Если был связанный `delegations/<slug>.md` item — переместить в `Done (7d)` того же файла.
4. Одна строка в `log.md`: `YYYY-MM-DD: ✅ RESOLVED Bx | <имя> → <что разблокировалось>`.
5. `tasks.md` **не трогается** — если блок разошёлся, следующий execution-шаг агент поставит в Active при обычном mid-flight sync.

Обратный порядок (сначала tasks.md) запрещён: он оставляет plan.md с устаревшим `blocked on Bx` и создаёт дрейф между быстрым и медленным слоем.

#### 5.4.3. Stale check как bootstrap-scan

Вместо weekly review блокеры проверяются при **каждом открытии проекта** (bootstrap) по триггеру «сколько дней висит». Это не отдельный ритуал, а часть нормального чтения `plan.md` в начале сессии.

Пороги:
- **>30 дней, тип `external-event`:** нормально, но агент показывает владельцу: «B1 висит 34 дня, ждём клиента — нужно пнуть или сменить fallback?»
- **>30 дней, тип `delegated-work`:** агент подтягивает `delegations/<slug>.md` и показывает статус связанной строки; если в трекере тоже нет движения — предлагает сменить fallback.
- **>60 дней, любой тип:** показывается как stale — агент спрашивает, не пора ли перевести в Resolved (через fallback) или закрыть milestone.
- **>90 дней, любой тип:** требует **решения** — либо снять блок, либо принять как постоянное ограничение и переделать план (milestone уходит в Contingency или Non-goals).

Поле `Stale check` в карточке блокера переопределяет дефолтные пороги, если для конкретного блокера они не подходят (например, «check through 14d — быстрый цикл»).

#### 5.4.4. Что НЕ является блокером

- **⏹️ STOPPED «закончился день».** Не блокер, а простая пауза — идёт только в `log.md`.
- **Шаг в `Next` с низким приоритетом.** Не блокер, просто не сейчас.
- **«Надо подумать».** Не блокер — это open question под milestone, правка `plan.md` в `Milestones` или `Drift Guard`.
- **💡 SPAWNED из прежней модели.** Категориальная ошибка: появление новой идеи — это не тип прерывания, а **событие маршрутизации**. Идея уходит в свой целевой файл (`plan.md` другого проекта, `personal/tasks.md`, `00_inbox/`, `delegations/`), а текущая задача продолжается без парковки.

## 6. Делегирования

### 6.1. Место

```
01_now/ops/<contour>/delegations/<person-slug>.md
```

Разделение по контурам: делегат одного продукта и делегат другого продукта не смешиваются. Карточка человека в `03_knowledge/contacts-network/persons/<person>/` остаётся описанием человека; файл делегирований — это рабочий реестр, у него другой жизненный цикл.

### 6.2. Принцип

Внешний таск-трекер контура — **source of truth** задачи. Файл в vault — **lean index**: одна строка на задачу, только ссылка на трекер, суть, даты, источник.

### 6.3. Формат строки

```markdown
- [ ] 2026-04-12 → обновить шаблон отчёта [TRACKER-1423](url) • due: 2026-04-20 • ссылка на файл встречи
```

Поля: статус, дата делегирования (= дата встречи), суть одной фразой, ссылка в трекер, `due:` дата обещания (опционально), ссылка на встречу-источник.

### 6.4. Секции файла

```
## Active      — в работе
## Done (7d)   — закрыто за последнюю неделю
## Archive     — всё старше 7 дней done
```

Три секции, не пять. `Waiting` и `Overdue` выражены через поле `due:` и grep (`rg "due: 2026-0[1-3]"` — найти просрочки). Меньше секций — меньше шанс, что структура рассыплется при неактивном weekly review (валидация 2026-04-15: пятиуровневая структура избыточна).

`Archive` растёт, но читается редко. За год ~100 строк — нормально.

### 6.5. Live-first audit

Статус задачи агент обновляет только по живому наблюдению в трекере (Правило live-first audit). Вне трекера статус не хранится, только дублируется для быстрого среза.

### 6.6. Passive recall при упоминании делегата

Weekly review — ритуал, которого у владельца нет (flow-worker профиль). Вместо него — **passive recall**: когда имя делегата всплывает в свежем контексте (встреча, чат, разбор), агент автоматически проверяет его `delegations/<slug>.md` и показывает открытые строки.

Триггеры passive recall:
- Скилл `meeting-processing` видит участника встречи → grep по `delegations/` всех контуров этого участника → show open.
- Любой запрос, содержащий имя делегата → тот же grep.
- Открытие проекта, в котором делегат упомянут → agent при bootstrap показывает его open delegations контура.

Это заменяет «я не помню, что делегировал полгода назад». Не идеально, но работает без weekly review.

## 7. Личные задачи

### 7.1. Место

```
01_now/personal/tasks.md
```

Один файл. Две секции. Никаких дополнительных файлов для идей/желаний.

### 7.2. Что попадает

Только **внешние обязательства миру**: сроки, документы, платежи, бюрократия, визиты к специалистам. Объём в нормальном состоянии — 5–15 записей.

### 7.3. Что НЕ попадает никогда

- Идеи и curiosity («почитать про X») → `00_inbox/` или `03_knowledge/`.
- Проектная работа («написать draft письма») → в проект.
- Желания («хочу заняться Y») → не записываются. Захочется — сделается.

Принцип: запись в этом файле определена через «придётся», а не через «хочу».

### 7.4. Формат

```markdown
---
режим: personal-hard-only
правило: только внешние обязательства, не желания
last_rewrite: 2026-04-15
next_rewrite_hint: 2026-05-15
---

# Personal Tasks

Last rewrite: 2026-04-15

## Hard (внешние обязательства)
- [ ] 2026-04-10 → продлить документ до 2026-06-01
- [ ] 2026-04-14 → подать отчётность до 2026-04-30

## Maybe (напоминания, могут быть удалены)
- [ ] 2026-04-08 → проверить новые фильтры в почте

## Done (последние 7 дней)
- [x] 2026-04-09 → записаться к стоматологу
```

Поле `last_rewrite` — визуальный маркер, что файл недавно «просыпан через сито», а не забыт (заимствование из Newport shutdown ritual, валидация 2026-04-15). Если `last_rewrite` старше месяца — файл считается несвежим, содержимому доверять нельзя без просмотра.

### 7.5. Governance by forced rewrite

Вместо TTL-таймеров — **принудительный rewrite раз в месяц**. По триггеру (`/personal rewrite` или первому ambient-упоминанию после истечения месяца) агент:
1. Читает файл целиком вслух владельцу — секция за секцией.
2. По каждой строке — один вопрос: «актуально?» → keep / delete / move.
3. Записывает свежий файл. `Done` очищается полностью. `Maybe`, которое не перекочевало в `Hard` — удаляется.
4. Обновляет `last_rewrite` в frontmatter и шапке.
5. Одна строка в `04_logs/personal.log`.

Принцип: файл не накапливает пыль, а регулярно переписывается с нуля на основе того, что ещё звенит. Это ближе к Newport-практике «небольшие списки, которые регулярно стираются», чем жёсткие TTL на каждую запись.

Если месяц не пройден, но файл разросся (>20 записей в Hard) — тоже триггер rewrite.

### 7.6. Дисциплина чтения

Файл **не читается на ежедневной основе**. Только по явному запросу («а что у меня висит?»). Если начать смотреть каждое утро — файл превратится в ещё один TickTick и провалится. Это сознательное проектное решение, не упущение.

### 7.7. Ambient capture

Когда в диалоге владелец вскользь упоминает обязательство — агент добавляет строку без MR-diff и без подтверждения. Строка лёгкая, удаляется по TTL, риск коллизий минимален. Если упоминание неоднозначно («может быть, надо X») — агент уточняет одним вопросом.

## 8. Эпизодические факты со встреч

### 8.1. Проблема

На встречах вскользь упоминаются факты, которые в момент встречи кажутся шумом, но через месяц становятся важными. Класть их в `context.md` проекта нельзя — он перестанет быть инвариантным. Терять нельзя.

### 8.2. Решение: hashtag RAG в саммари встреч

В файле саммари встречи (`materials/<contour>/meetings/YYYY-MM-DD_тема.md`) добавляется секция:

```markdown
## Упомянуто вскользь
- #onboarding: Таня сказала, что у них был провал с видеогайдами в 2025
- #pricing: Дима упомянул желание пересмотреть scale-тариф
- #hiring: был вопрос про Казанова, замяли
```

Агент при bootstrap проекта, если в запросе звучит тема `X`, делает:

```bash
grep -rh "#<X>" 01_now/projects/*/materials/**/meetings/
```

и подтягивает найденные фрагменты как дополнительный контекст. Полнотекстовый поиск по ~100 саммари в год — миллисекунды.

### 8.3. Canonical tag registry

Коллизии тегов (`#onboarding` vs `#activation`, `#hiring` vs `#recruiting`) — известная боль PKM. Решение: **canonical registry в `meta/TAGS.md`**, секция `## Task routing tags`.

Правила:
- Скилл `meeting-processing` при написании `## Упомянуто вскользь` использует **только canonical теги** из registry.
- Если нужен новый тег — скилл добавляет его в registry с одной строкой определения, затем использует.
- Синонимы не заводятся — первое назначенное имя остаётся канонiческим.
- При конфликте (старый файл с `#activation`, registry знает только `#onboarding`) — агент либо спрашивает, либо при касании старой встречи переводит тег в канонический.

### 8.4. Когда эпизодика поднимается в инвариант

Первичный триггер — **формулировка опиниона**, а не счётчик. Когда агент или владелец формулируют вывод по теме («мы знаем про X, что он Y») — это момент, когда факт готов переехать в `context.md` проекта как устойчивый инвариант.

Трёх упоминаний в разных встречах — эвристика, чтобы не пропустить момент. Когда она срабатывает, агент **не переносит автоматически**, а говорит: «тема #X упомянута в 3 встречах, готов ли сформулировать про неё опинион?». Если да — в `context.md`. Если нет — остаётся эпизодикой до следующего накопления.

Заимствование из Evergreen notes (Andy Matuschak): эвергрин живёт там, где есть strong opinion или strong fact, но не простой факт без формулировки.

### 8.5. Что это НЕ делает

- Не заменяет семантический поиск / embeddings — для объёма сотни файлов он не нужен.
- Не требует инфраструктуры — только grep и convention на хэштеги.
- Не хранит копию факта в нескольких местах — факт живёт только в саммари.

## 9. Контракт маршрутизации для скилла `meeting-processing`

Скилл заменяет текущую одномерную раскидку на вот этот routing:

| Слой из встречи | Куда идёт |
|---|---|
| Решение проекта | `log.md` проекта, 1 строка |
| Устойчивый инвариант | `context.md` проекта, только если проходит self-check «переживёт 5 встреч» |
| Стратегическая развилка / решение о направлении | `plan.md`, Milestones или Contingency |
| Action item агента по текущему milestone | `tasks.md` проекта, Next |
| Моё обязательство не по проекту | `01_now/personal/tasks.md`, Hard |
| Делегирование | `01_now/ops/<contour>/delegations/<person>.md`, Active |
| Эпизодический факт | саммари встречи, `## Упомянуто вскользь`, с хэштегом |
| Факт о людях и участии | `interactions.csv` (как сейчас) |

Перед каждой строкой скилл обязан запустить decision tree из §4.

## 10. Миграция существующих проектов

Прежние `tasks.md` проектов содержат смесь всех классов. Миграция — не автоматическая, она делается при первом касании проекта:

1. Агент при следующем открытии проекта читает текущий `tasks.md`.
2. Прогоняет decision tree §4 по каждой строке.
3. Показывает владельцу MR-diff: «это → plan.md», «это → personal/tasks.md», «это → delegations/<person>.md», «это → остаётся в tasks.md».
4. После подтверждения переносит и создаёт `plan.md`.
5. Записывает в `log.md` проекта одну строку: «миграция в новую модель маршрутизации по `task-routing-methodology-2026-04.md`».

Массовая миграция всех активных проектов одним махом запрещена — новые модели должны быть обкатаны на 1–2 проектах до раскатки.

## 11. Что этот документ НЕ решает

- Weekly review ритуал. Отдельный вопрос — как и когда владелец смотрит сводку по всем четырём файлам.
- Интеграцию с внешними трекерами задач. Сейчас — ручная ссылка в файле делегирований, автоматизация отдельной задачей.
- Governance для `backlog` проекта. Пока считаем, что backlog может оставаться в `plan.md` отдельной секцией.
- Миграцию старых саммари встреч в новую секцию `## Упомянуто вскользь` с хэштегами. Применяется только к новым встречам с момента принятия методологии.
