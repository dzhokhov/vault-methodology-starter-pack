# Методология происхождения знаний

## Суть

Любое важное знание в хранилище должно отвечать на вопрос: откуда оно взялось, насколько ему можно доверять и какие исходные материалы подтверждают вывод.

## Детали

### Зачем это нужно

Агент умеет быстро перерабатывать входящие материалы: встречи, заметки, исследования, таблицы, письма, документы. Без явной связи между исходником и выводом со временем становится непонятно:

- факт был сказан человеком или выведен агентом;
- это единичное наблюдение или устойчивый вывод;
- можно ли использовать вывод для решения;
- какой исходный материал нужно проверить при сомнении;
- не потерялась ли важная оговорка при пересказе.

Методология происхождения знаний нужна не для бюрократии, а для доверия к базе.

### Базовая цепочка

В простом случае знание проходит четыре слоя:

| Слой | Что это | Где хранить |
|---|---|---|
| Источник | Сырой материал: запись, расшифровка, файл, ссылка, заметка | `00_inbox/` или внешний источник |
| Извлечение | Краткий разбор источника: факты, решения, вопросы | рядом с проектом или в файле встречи |
| Устойчивый вывод | Обобщение, которое переживает источник | `context.md`, `03_knowledge/` или профильный файл |
| Использование | Решение, план, задача, скилл, шаблон | `plan.md`, `tasks.md`, `skills/`, `meta/templates/` |

Чем дальше слой от источника, тем важнее ссылка назад.

### Минимальные поля для важного знания

Для файлов, где важна проверяемость, добавляй в служебную разметку:

```yaml
source_type: meeting | file | research | manual_note | synthesis | imported
source_path: "путь/или/ссылка/на/источник"
derived_from:
  - "путь/к/исходнику.md"
confidence: single-source | emerging | confirmed | canonical
verification_status: unverified | in_review | verified_by_me | verified_by_team
```

Если файл уже использует `source_path` как путь самого файла в хранилище, исходники можно хранить в `derived_from`.

### Уровни доверия

| Уровень | Когда ставить | Что можно делать |
|---|---|---|
| `single-source` | Один источник, один человек, одна встреча | Хранить как наблюдение, не строить важное решение без проверки |
| `emerging` | Несколько независимых сигналов, но ещё мало данных | Использовать как гипотезу |
| `confirmed` | Подтверждено несколькими источниками или данными | Использовать как рабочий инвариант |
| `canonical` | Долго не опровергалось и стало частью системы | Использовать как устойчивое знание |

Уровень доверия можно повышать и понижать. Если появился контрпример, его нужно записать рядом с выводом.

### Правило для агента

Когда агент переносит знание из источника в устойчивый слой, он обязан:

1. Сохранить ссылку на источник.
2. Отделить факт от интерпретации.
3. Поставить консервативный уровень доверия.
4. Записать, что именно изменилось в `log.md`, если вывод влияет на проект.
5. Не удалять исходник без прямого решения владельца.

### Пример безопасной разметки

```yaml
id: customer-support-patterns
type: reference
status: active
created: 2026-04-30
updated: 2026-04-30
aliases:
  - "Повторяющиеся обращения поддержки"
tags: [support, patterns]
source_path: "03_knowledge/customer-support-patterns.md"
source_type: synthesis
derived_from:
  - "01_now/projects/example/meetings/meeting-2026-04-20-support.md"
  - "00_inbox/support-export-2026-04.csv"
confidence: emerging
verification_status: in_review
```

### Что не делать

- Не писать «известно, что...» без ссылки на источник.
- Не повышать доверие только потому, что текст звучит убедительно.
- Не смешивать в одном абзаце цитату, пересказ и вывод агента без пометок.
- Не делать рабочий проект долговременным домом для знаний, которые переживут проект.
- Не хранить чувствительные исходники в общем пакете только ради проверяемости.
