# AGENTS.md — правила агента для файлового хранилища

## Главный принцип

Это хранилище работает только тогда, когда до каждого важного документа можно добраться через указатели, ссылки и правила маршрутизации.

Если агент создаёт или меняет файл, он обязан обновить связанные индексы и кратко зафиксировать решение в нужном журнале.

## Карта папок

| Папка | Назначение |
|---|---|
| `00_inbox/` | Новые и ещё неразобранные материалы |
| `01_now/` | Активная работа: проекты, текущие области, личные задачи |
| `02_domains/` | Постоянные области жизни или бизнеса |
| `03_knowledge/` | Переиспользуемые знания, методологии, справка |
| `04_logs/` | Хронология, обзоры, журналы решений |
| `90_archive/` | Завершённое и устаревшее |
| `meta/` | Правила, шаблоны, служебные индексы |
| `skills/` | Скиллы: инструкции для повторяющихся типов задач |

## Загрузка контекста

Перед ответом агент определяет тип задачи и читает минимальный нужный контекст:

| Тип задачи | Что читать |
|---|---|
| Активный проект | `01_now/projects/<project>/README.md`, затем `context.md`, при необходимости `plan.md`, `tasks.md`, `log.md` |
| Постоянная область | `02_domains/README.md`, затем профильный раздел |
| Справка или методология | `03_knowledge/README.md`, затем нужный файл |
| Запись или изменение файла | `meta/rules/write-protocol.md` |
| Маршрутизация задач | `meta/rules/task-routing.md` |
| Обработка входящих материалов | `meta/rules/inbox-processing.md` |
| Проверка и уборка хранилища | `meta/rules/vault-review.md` |
| Создание временных файлов | `meta/rules/cleanup-by-design.md` |

Если контур или проект неоднозначен, агент задаёт один короткий вопрос на уточнение и не читает содержательные файлы разных контуров подряд.

## Создание проекта

Новый активный проект создаётся в `01_now/projects/<год>-<slug>/`.

В проекте должны быть:

- `README.md` — вход, смысл и навигация;
- `plan.md` — цель, границы, вехи, блокеры;
- `tasks.md` — текущая очередь исполнения;
- `context.md` — устойчивые знания проекта;
- `log.md` — короткая история решений.

`plan.md` хранит направление и вехи. `tasks.md` хранит только текущие действия. История не живёт в `tasks.md`; она живёт в `log.md`.

## После записи файла

После создания или существенного изменения файла агент проверяет:

1. Есть ли корректная служебная разметка в начале `.md` файла.
2. Обновлён ли `README.md` раздела.
3. Нужно ли обновить `plan.md`, `context.md`, `log.md` или `tasks.md`.
4. Работают ли относительные ссылки.
5. Найдёт ли новый агент этот файл через навигацию.
6. Есть ли план жизненного цикла для временных файлов.

## Скиллы

Скиллы лежат в `skills/<name>/SKILL.md`. Агент читает скилл только тогда, когда задача подходит под его описание.

Базовые скиллы:

- `meeting-processing` — обработка встреч и расшифровок;
- `research` — исследование и фиксация результата;
- `ilyakhov-editor` — редактура текста;
- `parking` — сохранение точки возврата;
- `resume` — возврат к сохранённой задаче;
- `new-dialog-handoff` — безопасный переход в новый чат;
- `owner-only-dev-orchestrator` — ведение задач разработки с минимальным участием владельца;
- `test-gates` — проверка изменений;
- `release-rollback` — выпуск и откат.

## Язык ответа

Агент отвечает на языке пользователя. Если использует технический термин, он объясняет его простыми словами при первом упоминании.

## Запреты

- Не создавать файлы без понятного места и назначения.
- Не оставлять важные документы без ссылки из указателей.
- Не хранить долгоживущие знания внутри временного проекта.
- Не складывать личные обязательства, делегирования и цели проекта в один список.
- Не удалять логи и входящие материалы без явной просьбы владельца.
- Не читать личную почту, мессенджеры или внешние сервисы без прямой просьбы.

## Следующий шаг

Если это первый запуск, агент должен прочитать `START_HERE.md`, этот файл и `meta/rules/write-protocol.md`, затем кратко объяснить владельцу, как устроено хранилище.
