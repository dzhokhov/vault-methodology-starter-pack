# Cleanup by Design: правило жизненного цикла артефактов

## Суть

Любой агент, создающий файл, базу данных, кэш или дамп — **обязан** в том же действии определить, как и когда артефакт будет очищен. Без lifecycle-плана создание артефакта **запрещено**.

Vault — рабочая система, а не свалка. Каждый артефакт либо живёт осознанно, либо не создаётся.

---

## Что НЕ удалять автоматически

- **Логи** (`*.log`, `runs.log`) — показывают состояние проекта, историю запусков, ошибки. Удалять только по явной команде владельца.
- **Содержимое `00_inbox/`** — это зона разбора, не мусор. Файлы могут быть ценными. Разбирается вручную через [vault-review](./vault-review.md).
- **Содержимое `01_now/`, `02_domains/`, `03_knowledge/`** — permanent хранилище. Архивируется, не удаляется.
- **Дампы данных** (`*-dumps/`) — могут понадобиться для анализа. Удалять только по решению владельца.
- **Файлы делегирований** (`01_now/ops/<contour>/delegations/<person-slug>.md`) — НЕ удалять автоматически даже если все строки переехали в `Archive`. Ротация — через forced rewrite владельцем (см. [vault-lifecycle-sop.md](../vault-lifecycle-sop.md#очистка-делегирований)).
- **`01_now/personal/tasks.md`** — НЕ удалять и не архивировать автоматически. Этот файл ротируется только владельцем через monthly forced rewrite.

---

## Что МОЖНО удалять автоматически

| Класс | Где | TTL | Почему безопасно |
|-------|-----|-----|-----------------|
| **temp** | `tmp/` | 7 дней | Промежуточные файлы, тесты, build-артефакты. Всё ценное уже извлечено. |
| **cache** | `.smart-env/multi/` | 30 дней | Obsidian Smart Connections, регенерируется автоматически |
| **cache** | `*/.model-cache/` | 30 дней | ML-модели, скачаются заново |
| **db records** | SQLite: `classification='trash'` | 90 дней | Email metadata с классификацией «мусор» |
| **db records** | SQLite: `classification='fyi'` | 180 дней | Email metadata «к сведению» |

---

## Обязательные действия агента при создании артефакта

### 1. Определи класс

Спроси себя: «Этот артефакт будет нужен через месяц?»
- Да → permanent (не автоочистка)
- Нет → определи TTL и способ очистки

### 2. Если артефакт НЕ permanent — пропиши в коде

```python
# LIFECYCLE: temp, TTL=7d, cleanup via vault-hygiene.py
```

В каждом скрипте, создающем временные файлы или записи с TTL, должна быть либо:
- функция очистки в самом скрипте, либо
- запись в `meta/cleanup-registry.md` чтобы `vault-hygiene.py` знал про этот артефакт

### 3. Если создаёшь новую папку

Добавь запись в [реестр артефактов](../cleanup-registry.md).

### 4. Если создаёшь SQLite

Обязательно:
- DDL с датой создания записи (`created_at` / `fetched_at`), чтобы было поле для TTL-фильтрации
- Зарегистрировать базу в [cleanup-registry.md](../cleanup-registry.md): путь, таблицы, TTL записей, кто создаёт
- Определить стратегию очистки для каждой таблицы:

| Тип данных | Стратегия |
|------------|-----------|
| Кэш, промежуточные данные | DELETE по TTL + VACUUM |
| Метаданные с классификацией | DELETE по classification + TTL (как email: trash=90д, fyi=180д) |
| Рабочие записи (actionable) | Не удалять автоматически, только по решению владельца |
| Лог обработки (processed) | Хранить ∞ — это аудит-трейл |

### 5. Мониторинг SQLite

Скрипт `vault-hygiene.py` в режиме `--report` показывает для каждой зарегистрированной базы:
- Размер файла
- Количество записей по таблицам
- Количество expired записей (готовых к очистке)
- Дату последней записи (свежесть данных)

---

## Запрещено

- ❌ Создавать файлы без понимания их lifecycle
- ❌ Создавать SQLite-базы без поля даты для TTL
- ❌ Автоматически удалять содержимое `00_inbox/` (это зона разбора, не мусор)
- ❌ Автоматически удалять логи (они показывают состояние проекта)
- ❌ Предполагать, что владелец сам разберётся с очисткой
- ❌ Создавать артефакт, не проверив, нет ли уже подобного (анти-дублирование)

---

## Инструменты

- **Автоочистка:** `scripts/vault-hygiene.py` — чистит только safe-to-delete (tmp/, кэши, expired DB records)
- **Отчёт:** `python3 vault-hygiene.py` (без --clean) — показывает дисковое использование и что можно почистить
- **Ручное ревью:** [vault-review.md](./vault-review.md) — разбор inbox, архивация завершённых проектов
- **Реестр:** [cleanup-registry.md](../cleanup-registry.md) — что где с каким TTL
