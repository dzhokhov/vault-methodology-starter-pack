# Протокол разработки и скриптов

## Суть
Правила для любых задач, где агент создаёт или изменяет код, скрипты, программы. Минимум участия пользователя, обязательная самопроверка, итоговый статус GREEN/YELLOW/RED.

## 1. Протокол скриптов

При создании любого скрипта:
1. **Сначала объясни** логику, ограничения и варианты реализации
2. **Пиши код только после подтверждения**
3. **После запуска** — запиши результат и команду отката в `scripts/runs.log`

Подробности: [Протокол создания и применения скриптов](../script-protocol.md)

## 2. Owner-only разработка

Для любой задачи с изменением кода:

1. Работать по стандарту: [Owner-only стандарт разработки](../../03_knowledge/ai/owner-only-agent-dev-standard-2026-03.md)
2. Минимизировать участие пользователя: пользователь задаёт цель и принимает только релизное решение; агент выполняет реализацию и полную самопроверку
3. Перед изменениями обеспечить процесс-артефакты:
   - `AGENT_WORKFLOW.md`, `TEST_PLAN.md`, `RELEASE_CHECKLIST.md`, `CHANGELOG.md`, `INCIDENT_RUNBOOK.md`
   - Если нет — создать из шаблонов `meta/templates/`
4. Перед статусом «готово» пройти quality gates: lint/type/static, unit/integration, smoke, rollback-план
5. Отчёт только в формате:
   - `GREEN` — все проверки пройдены, релиз безопасен
   - `YELLOW` — есть риски или неполные проверки
   - `RED` — выпуск запрещён, требуется доработка

## 3. Общий стандарт agentic-разработки

- Глобальные правила — короткими, проектную специфику — рядом с кодом
- Перед кодом — обязательный `ask/plan`
- Работу разбивать на milestones
- `tasks.md` для кодовых задач вести только в режиме `development`: один `Active Step`, обязательные `Exit Criteria` и `Drift Guard`
- Не допускать silent fallback на критическом пути
- Неоднозначные автоматические решения → review queue

Полный reference: [Общие правила agentic-разработки](../../03_knowledge/ai/general-rules-for-agentic-development-in-local-vault-2026-03.md)

## 4. Правила выдачи терминальных команд

- Перед **любой** терминальной командой проверить пути: рабочая папка пользователя ≠ VM-путь. Указывать полные абсолютные пути
- Перед `rm`, `mv`, `sed -i`, `git reset`, `DROP` — показать что будет удалено/перемещено/изменено и получить подтверждение
- Все скрипты — через `python3`, не `python`. Для pip: `--break-system-packages`
- При ошибке `Permission denied` → не делать `chmod 777`, предложить альтернативу

## 5. Lifecycle артефактов кода (ОБЯЗАТЕЛЬНО)

Любой скрипт, который **создаёт файлы, базы данных, кэши или дампы**, должен:
1. Иметь встроенный механизм очистки (параметр `--clean`, TTL-логика, ротация)
2. Документировать что создаёт и где — в комментарии в коде + в [cleanup-registry.md](../cleanup-registry.md)
3. Для SQLite: обязательно поле даты (`created_at`/`fetched_at`) для TTL-фильтрации

Если скрипт создаёт данные, но не предусматривает как их удалять — это баг, а не фича.

Полные правила: [Cleanup by Design](./cleanup-by-design.md)

## 6. Обновление вместо костылей

При обнаружении бага или несовместимости: исправить **причину** (обновить формат, починить генератор, исправить схему), а не обходить симптом. «Обходной путь» допустим только как временная мера с TODO-маркером и задачей на исправление.
