# Правило: Live-first audit

## Суть
Текущий статус любого внешнего объекта (рекламная кампания, CRM-воронка, сайт, автоматизация, интеграция) агент обязан подтверждать наблюдением на живом источнике. Лог — это история действий, а не текущее состояние.

## Детали

### Почему
С объектами работают руками. Настройки меняются между сессиями, маркетолог/владелец/оператор могут поправить что угодно, и в `log.md` эта правка может не попасть. Опора на лог как на источник текущего статуса = ошибка аудита и ложные выводы.

### Правило
1. **Аудит = наблюдение.** Перед тем как сказать «кампания в таком-то состоянии», «URL такой-то», «такое-то объявление активно/приостановлено», агент обязан получить это значение из живого источника: API, UI в браузере, сам сайт, админка CRM, dashboard аналитики.
2. **Лог — только для истории.** `log.md`, `tasks.md`, прошлые записи в `context.md` допустимы только как источник:
   - истории действий (что и когда делалось);
   - известных блокеров (чтобы не наступать снова);
   - гипотез о причине расхождений.
   Они **не** являются источником текущего статуса.
3. **Сравнение обязательно.** После наблюдения агент сравнивает реальное состояние с тем, что зафиксировано в логе:
   - что совпало → подтверждено;
   - что не совпало → отдельный блок «расхождения», с датой наблюдения;
   - что не могло быть записано (новые правки руками) → отдельный блок «новые изменения», записать в `log.md` этой же сессией.
4. **Если живой источник недоступен — сказать это явно.** Формулировки «по логу на такую-то дату» и «наблюдал живьём такого-то числа» не взаимозаменяемы. Если API/браузер/живой доступ заблокирован (rate limit, нет токена, связь), агент обязан сказать: «Живой аудит недоступен по причине X. Ниже — последнее известное состояние по логу от DATE, точность не гарантирована.» И не выдавать это за текущий статус.
5. **Порядок действий при запросе «проверь X».**
   1. Попытаться снять состояние с живого источника.
   2. Если получилось → сверить с логом, обновить log/context при расхождении.
   3. Если не получилось → явно сказать об этом, попросить владельца либо починить доступ, либо разрешить отвечать по логу как по истории.
   Запрещено: молча отвечать по логу, выдавая это за текущий статус.

### Применимость
- Рекламные кампании (Meta, Google, TikTok, Yandex): API → Ads Manager UI.
- CRM: API → интерфейс.
- Сайты и лендинги: прямой HTTP-запрос, не описание из ТЗ.
- Интеграции (Pixel, CAPI, webhook, scheduler): проверка конечной точки, а не конфига из README.
- Аналитика (GA4, Metabase): свежая выгрузка, а не прошлый CSV.
- **Делегирования сотрудникам**: внешний трекер = источник правды. Файл `01_now/ops/<contour>/delegations/<person-slug>.md` — это краткий индекс, а не источник статуса задачи. Если владелец спрашивает «что у ответственного?» — сначала наблюдение в трекере по ссылкам из файла, только потом ответ. Если трекер недоступен — явно помечать «по файлу, без живой проверки». См. [task-routing.md §delegations](./task-routing.md).

### Что делать с расхождениями
Если живое состояние расходится с логом:
1. Записать расхождение в `log.md` этой же сессии отдельной записью: `наблюдение → ожидание по логу → дельта → гипотеза причины`.
2. Обновить `context.md`, только если расхождение отражает устойчивое изменение состояния (не разовую правку руками в UI).
3. Никогда не править историю в `log.md` задним числом — добавлять новую запись.

### Исключения
Только одно: если владелец явно говорит «отвечай по логу, живьём не ходи» — тогда отвечать по логу, но явно пометить ответ как `по-логу, не-live`.
