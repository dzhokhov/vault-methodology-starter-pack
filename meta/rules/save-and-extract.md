# Порог сохранения и экстракция знаний

## Суть
Когда сохранять результаты в vault, когда нет. Как экстрагировать переиспользуемые знания из проектов.

## Тройной фильтр: сохранять ли результат

После завершения задачи оцени по трём критериям:

1. **Переиспользуемость.** Полезно в будущих сессиях? (Словарь — да. «Какая погода» — нет.)
2. **Уникальность.** Содержит данные, которых нет в базе? (Анализ бюджета — да. Пересказ документа — нет.)
3. **Объём.** Содержательнее 1 абзаца? (2 строки → не создавать файл.)

**Хотя бы 2 из 3** → предложи конкретное действие: «Сохранить в `03_knowledge/adtech-market-2026.md`?»
**Ни один** → не сохранять, не спрашивать.

## Ресёрч и best practices — ВСЕГДА сохранять

Когда пользователь просит провести ресёрч, собрать best practices, изучить подходы:

1. Сохранить результат в `03_knowledge/`. Если подходящего домена нет — создать (с учётом анти-фрагментации).
2. Формат: полноценная заметка с YAML, `## Суть`, **ссылки на источники** (URL, авторы). Без источников ресёрч теряет доверие.
3. Связать с проектом: если ресёрч в контексте проекта — ссылка из `context.md`.
4. Не спрашивать «сохранить?» — сохранять всегда. Ресёрч по определению проходит тройной фильтр.
5. Без явного проекта → предложить создать проект или standalone-заметку.

**Мотивация:** собранная информация — самый ценный побочный продукт работы с LLM. Потеря ресёрча из-за завершения чата — непростительна.

## Экстракция знаний из проектов

Когда в проекте создаётся документ, полезный **за пределами** проекта:
- Основной файл → в `03_knowledge/` или `02_domains/`
- В проекте → ссылка из context.md

Примеры: словарь терминов → `03_knowledge/`, процедура → `03_knowledge/`, контакты и роли → `02_domains/work/`.
