# Протокол создания и изменения файлов в vault

## Суть
Единый свод правил при записи: frontmatter, структура, ссылки, обновление индексов, анти-фрагментация. Применяется при создании или редактировании **любого** файла vault.

## 1. YAML frontmatter (ОБЯЗАТЕЛЬНО)

Каждый markdown-файл должен иметь:

```yaml
id: уникальный-slug
type: note | project | log | tasks | registry | reference | plan | index | rules
status: active | draft | archived
created: YYYY-MM-DD
updated: YYYY-MM-DD
aliases:
  - "Короткое русское название"  # минимум 1, различимое на уровне vault
tags: [релевантные, теги]
source_path: "путь/от/корня/vault.md"
```

Необязательные поля:
```yaml
freshness: evergreen | seasonal | ephemeral  # по умолчанию seasonal
expires: YYYY-MM-DD  # дата ревью
knowledge_criticality: low | medium | high | critical
verification_status: unverified | in_review | verified_by_me | verified_by_team
curation_mode: none | manual_edit | llm_explicit_request | imported_not_curated
```

`verification_status: verified_by_me` ставится **только после ручной проверки пользователем**. Для импортов по умолчанию: `unverified` + `imported_not_curated`.

### 1.1. Confidence ladder для полевых наблюдений (ОБЯЗАТЕЛЬНО)

Наблюдения из встреч, интервью и полевых исследований маркируются полем `confidence` в frontmatter. Это **не** редакторская проверка (`verification_status`) — это степень подтверждённости наблюдения независимыми источниками.

```yaml
confidence: single-source | emerging | confirmed | canonical
```

| Уровень | Критерий | Что можно делать |
|---------|----------|-----------------|
| `single-source` | Одна встреча / один человек | Фиксировать как field note. НЕ ссылаться как на факт в решениях. |
| `emerging` | 2-3 независимых источника | Формулировать гипотезу. Ссылаться с оговоркой «emerging pattern». |
| `confirmed` | 4+ источника ИЛИ разнотипные (встреча + данные + ресёрч) | Использовать как рабочий инвариант в context.md. |
| `canonical` | Подтверждено, не опровергнуто ≥3 месяцев | Domain knowledge. Строить продуктовые решения. |

**Повышение:** при появлении нового подтверждения — обновить `confidence`, добавить ссылку на источник, записать дату.
**Понижение:** при обнаружении контрпримера — понизить `confidence`, зафиксировать контрпример.
**По умолчанию:** любое наблюдение из одной встречи получает `single-source`. Без исключений.

Применяется к: `field-notes-*.md`, наблюдениям в карточках контактов, гипотезам в research-файлах.
Не применяется к: фактическим данным (даты, цифры, цитаты), проектным решениям, правилам vault.

Методология: [research-systematic-field-notes-methodology.md](../../03_knowledge/research-systematic-field-notes-methodology.md)

## 2. Рекомендуемая структура тела

- `## Суть` — одно предложение: что это и зачем (для LLM-поиска)
- `## Детали` — основной контент
- `## Следующий шаг` — если применимо

**Тест самодостаточности:** «Если LLM прочитает только этот файл — поймёт ли, о чём он и кому полезен?» Если нет — файл неполный.

## 3. Стандарт ссылок в vault (Obsidian)

- Только кликабельные Markdown-ссылки: `[Понятный анкор](./relative-path.md)`
- Анкор — человекочитаемое описание, не имя файла
- Относительные пути (`./`, `../`) с расширением `.md`
- Запрещено: путь в бэктиках как ссылка, анкоры-имена файлов (`[context.md](...)`), голые пути

### Верификация относительных путей (КРИТИЧЕСКИ ВАЖНО)

Перед записью ссылки — посчитать глубину:
1. Определить уровень файла-источника от корня vault (по `/`)
2. Каждый `../` поднимает на один уровень
3. После всех `../` должен быть корень vault, затем абсолютный путь

Пример: файл на глубине 4 (`01_now/projects/foo/tasks/task.md`) → ссылка на `00_inbox/bar.md` требует **4** уровня подъёма: `../../../../00_inbox/bar.md`.

**Правило:** после написания ссылки мысленно разрешить путь. Если используешь скрипт — включить `os.path.exists()`.

## 4. Ссылки в диалоге (Cowork / Claude Code)

Любой файл в ответе пользователю — кликабельная ссылка `computer://`:
- `[Понятное название](computer:///sessions/.../path/to/file.ext)`
- Не оставлять голые пути в бэктиках

## 5. Обязательный протокол после записи

После создания или существенного изменения файла — выполнить **все применимые шаги** в указанном порядке (медленные слои → быстрые слои):

1. **Обновить `plan.md` проекта** (если изменился Goal, Milestone, Appetite, Drift Guard, Contingency или Acceptance). `plan.md` — медленный контракт проекта; его правят до того, как что-то потечёт в `context.md` или `tasks.md`. См. [Правило 12 в AGENTS.md](../../AGENTS.md).
2. **Обновить `context.md` проекта** — только при появлении устойчивого инварианта (термин, метрика, SoT, ограничение). Эпизодика со встречи сюда НЕ попадает — она в `<meeting>.md` под `## Упомянуто вскользь` с canonical-тегом. См. [task-routing.md](./task-routing.md).
3. **Запись в `log.md` проекта.** Короткая запись: дата, что произошло, какое решение, ссылка на артефакт. Максимум 3–7 буллетов. Подробный контент — в отдельный артефакт, не в log.md. См. [AGENTS.md Правило 5](../../AGENTS.md).
4. **Обновить `tasks.md`** — только execution-слой (Active/Next/Waiting). Размышление («понять, что делать с X») идёт не сюда, а в `plan.md` под соответствующий milestone. Goal, Milestone, Contingency в `tasks.md` **запрещены**.
5. **Обновить делегирования** — если решение/шаг касается делегата, обновить строку в `01_now/ops/<contour>/delegations/<person-slug>.md` (одна строка + ссылка на внешний трекер как SoT). Если в контуре уже действует внешний трекер как живой рабочий слой, не заводить второй живой слой в общем markdown-контуре: обновлять внешний трекер, а `delegations/` трогать только если это краткий индекс или архивный переходный слой. См. [task-routing.md §delegations](./task-routing.md).
6. **Обновить личный `01_now/personal/tasks.md`** — если возникло внешнее обязательство владельца, не связанное с проектом. Ambient capture без MR-diff.
7. **Обновить `README.md` раздела** куда попал файл. Критерий: прочитав README, LLM узнает о существовании файла.
8. **Перекрёстные ссылки** — проверить обратные ссылки в связанных документах.
9. **Обновить `01_now/README.md`** — только если проект новый или закрыт.
10. **Самопроверка маршрутизации:** «Если завтра новая LLM-сессия спросит по теме документа — найдёт ли его через цепочку README → plan.md → context.md → файл?» Нет → пропущен шаг.
11. **Очистка временных артефактов** — удалить промежуточные файлы после интеграции знаний.

**Порядок критичен:** `plan` → `context` → `log` → `tasks` → `delegations/personal` → `README`. Медленные слои всегда опережают быстрые. Нарушение порядка = дрейф плана поверх старого контракта.

## 5.1. Mid-flight sync для project files

Не жди конца задачи, если уже возникло состояние, которое будет трудно восстановить из чата.

### Когда обновлять `log.md`

`log.md` обновляется **сразу по событию**, а не только при финализации:
- принято решение;
- завершён milestone;
- найден blocker;
- изменён план/подход;
- создан важный артефакт или закрыт заметный риск.

`log.md` отвечает на вопрос: **что произошло и почему**.

### Когда обновлять `context.md`

`context.md` обновляется только при появлении **устойчивого знания**, полезного будущим сессиям:
- новый термин, инвариант, ограничение;
- метрика успеха, критерий качества;
- source-of-truth документ или обязательная зависимость;
- новая стабильная договорённость о том, как устроен проект.

`context.md` отвечает на вопрос: **что теперь считается устойчиво верным про проект**.

### Чего не делать

- Не превращать `log.md` в backlog.
- Не писать в `context.md` временные повороты и мелкую хронологию.
- Не держать несколько значимых решений/находок только в чате до конца длинного цикла.

### Обязательный sync-триггер

Если следующий шаг нельзя надёжно восстановить по файлам проекта, или в чате уже накопились несколько несохранённых решений/находок, агент обязан сначала обновить `log.md` и/или `context.md`, а потом продолжать работу.

## 5.2. `plan.md` + `tasks.md`: контракт и execution-очередь

С апреля 2026 любой активный проект обязан иметь два файла:

- **`plan.md`** — медленный контракт проекта. Goal, Non-goals, Appetite, Source of truth, Milestones (с Acceptance), Drift Guard, Contingency. Меняется, когда меняется цель или границы.
- **`tasks.md`** — быстрая execution-очередь. Только Active/Next/Waiting шаги агента в текущем milestone.

Полная методология: [task-routing-methodology-2026-04.md](../../03_knowledge/task-routing-methodology-2026-04.md). Тактический чеклист: [task-routing.md](./task-routing.md). Правила владения: [AGENTS.md Правила 11–14](../../AGENTS.md).

### Режим проекта

Перед глубокой работой агент обязан выбрать один режим (указывается в `plan.md`, не в `tasks.md`):

- `operational` — обычные задачи без изменения исполняемой системы;
- `development` — код, скрипты, тесты, schema/data contracts, runtime, CI, build/release.

### Обязательные правила для `tasks.md`

- В файле явно укажи `Task Mode: operational | development`.
- Держи только один `Active` / `Active Step`.
- Если собираешься работать вне `Active` или `Next`, сначала обнови `plan.md` (если меняется вектор) или `tasks.md` (если только очередь).
- Не копи длинный список completed items: история должна жить в `log.md`.
- **Запрет:** `Goal`, `Milestones`, `Contingency`, `Appetite` — всё это в `plan.md`, не в `tasks.md`.
- **Запрет:** строки вида «понять, что делать с X», «разобраться с Y» — это не execution, это размышление; его место в соответствующем milestone в `plan.md` как открытый вопрос.
- **Запрет:** делегирование сотруднику («Дима сделает X») — это в `delegations/<person-slug>.md`. Личные обязательства — в `01_now/personal/tasks.md`.

### Структура `tasks.md` для `operational`

Минимум: `Active`, `Next`, `Waiting`, `Backlog`. Без `Goal` — он в `plan.md`.

### Структура `tasks.md` для `development`

Минимум: `Active Step`, `Exit Criteria`, `Drift Guard (short)`, `Next`, `Backlog`. `Current Milestone` ссылается на соответствующий пункт в `plan.md`, не дублирует его. Блокеры — в `plan.md §Blockers` как first-class сущности, секции `Blocked` в `tasks.md` нет.

### Drift protocol (обязательный)

Если агент меняет подход, scope, порядок реализации или обнаруживает, что текущий milestone больше не ведёт к цели:
1. Сначала правка **`plan.md`** (меняется Milestone, Acceptance, Drift Guard, Contingency или Non-goals).
2. Одна строка в `log.md`: «дрейф: было X, стало Y, причина Z».
3. Только потом обновление `tasks.md` и возобновление работы.

Нарушение порядка = контракт меняется под прикрытием execution'а, и будущий агент не найдёт следа решения.

## 6. Анти-фрагментация папок

- Сначала использовать существующую папку. Новая — только если иначе теряется группировка.
- Запрещено: подпапка ради 1-2 файлов.
- Новая подпапка: минимум 5+ файлов ожидается или долгоживущий контур.
- Глубина: не глубже 3-4 уровней от логического корня.
- Серийные файлы — порядок в имени (`YYYY-MM-DD_тип-номер.md`), не в структуре папок.

## 7. Жизненный цикл файлов

- Неясно куда → `00_inbox/`
- Разбор: активная работа → `01_now/`, область → `02_domains/`, справка → `03_knowledge/`, дневник → `04_logs/`, устаревшее → `90_archive/`
- `01_now` хранит только операционное (1-4 недели). Справочное → экстрагировать в `03_knowledge/`.
- Регламент: [Vault Lifecycle SOP](../vault-lifecycle-sop.md)

## 8. Lifecycle артефактов (ОБЯЗАТЕЛЬНО)

При создании **любого** нового файла, БД, кэша, дампа — сразу ответь:
- **TTL:** когда это можно удалить? (7д / 30д / 90д / вручную / никогда)
- **Очистка:** кто и как удалит? (скрипт vault-hygiene / ручной vault review / cron)
- **Регистрация:** зарегистрируй в [cleanup-registry.md](../cleanup-registry.md) если это не одноразовый файл

Если файл из `00_inbox/` — после обработки запиши в `00_inbox/PROCESSING_LOG.md`:
что обработано, куда перенесено, статус (✅ processed / 🔄 partial).
Подробности: [Протокол обработки inbox](./inbox-processing.md)

Полные правила: [Cleanup by Design](./cleanup-by-design.md)

## 9. Удаление и архивация

- При перемещении в `90_archive/` — удалить/пометить `[archived]` ссылки в README и context.md.
- При удалении файла — очистить все ссылки на него.

## 10. Интеграция знаний из временных источников

- В итоговых документах нельзя оставлять ссылки на временные файлы.
- Временные источники — одноразовый материал для синтеза.
- После интеграции — удалить промежуточные файлы, проверить отсутствие ссылок на них.

## 11. Обратная синхронизация после миграций

После **любой** миграции или изменения формата данных:
1. Найти все скрипты, шаблоны, регламенты, генерирующие затронутые данные.
2. Обновить каждый, чтобы он создавал данные в новом формате.
3. Тест: «Новый агент запускает импорт с нуля — получит новый формат?» Нет → миграция не завершена.
