# Протокол создания и применения скриптов

## Статус: ДЕЙСТВУЕТ С 2026-03-10

## ДО написания кода

1. **Объясни что делает скрипт** — 3–5 предложений, без кода. Вход, выход, какие файлы читает/пишет.
2. **Назови ограничения** — когда метод может дать неверный результат. Пример: «regex-парсинг YAML сломается на многострочных значениях».
3. **Предложи варианты** — если есть больше одного подхода, кратко опиши каждый с плюсами/минусами.
4. **Жди подтверждения.** Код — только после «да».

## ДО запуска

1. `--dry-run` (если поддерживается) — показать что изменится.
2. Бекап затрагиваемых файлов.

## ДОПОЛНИТЕЛЬНО ДЛЯ AGENTIC-СКРИПТОВ

Если скрипт:
- запускается в цикле `plan -> run -> test -> retry`,
- используется ИИ-агентом для автономной работы,
- или сам управляет многошаговым агентным сценарием,

то обычного `stdout/stderr` недостаточно. Такой скрипт обязан иметь **agent-readable logging contract**.

### Минимальный контракт логирования

Лог должен по возможности содержать:
- `run_id`
- `step_id`
- `file_or_module`
- `function_id`
- `block_id`
- `belief_state` — какую гипотезу агент сейчас считает верной
- `observed_result` — что реально произошло
- `next_action` — что агент собирается делать дальше

### Обязательные свойства agent-readable логов

1. **Log-to-code correlation**  
   В логах и коде должны повторяться одинаковые идентификаторы функций/блоков, чтобы агент мог склеивать код и лог в единый контекст.

2. **Structured format over prose**  
   Предпочтительны структурные поля, теги или повторяемые якорные токены вместо свободного текста без разметки.

3. **Belief-state logging**  
   Агент должен фиксировать не только ошибку, но и свою текущую гипотезу о причине и механике сбоя.

4. **Anti-loop protection**  
   Для повторных неудач должны быть retry limit, явная остановка или смена стратегии. Бесконечный retry без протокола остановки запрещён.

5. **Forced context on repeated failure**  
   Если сценарий падает повторно, агенту должен подаваться summary прошлых неудачных попыток или known-fix context, а не только голая ошибка текущего прогона.

## ПОСЛЕ запуска

1. Показать результат: сколько файлов создано/изменено/удалено, ошибки.
2. Записать в `scripts/runs.log`:

```
YYYY-MM-DD HH:MM | script_name.py | параметры
  Результат: N создано, M изменено, K ошибок
  Бекап: путь
  Откат: команда для отката
```

## Лог применения

Файл `scripts/runs.log` — обязательный журнал всех запусков. Без записи в лог запуск скрипта, изменяющего данные, запрещён.

Для agentic-скриптов `runs.log` не заменяет runtime-логи, а только дополняет их как журнал факта запуска.
