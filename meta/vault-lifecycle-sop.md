# Vault Lifecycle SOP (now -> knowledge -> archive)

## Суть
Операционный регламент, который держит `01_now` только для текущей работы, а стабильные знания переносит в `03_knowledge`/`02_domains`, с регулярной архивизацией и очисткой временных артефактов.

## Слои хранения
- `00_inbox` — входящий и неразобранный поток.
- `01_now` — только активная работа и текущие проекты.
- `02_domains` — постоянный контекст роли/сферы (долгоживущий фон).
- `03_knowledge` — переиспользуемые знания вне конкретного проекта.
- `04_logs` — хронология и weekly/monthly review.
- `90_archive` — завершенное и historical/legacy.

## Критерии «оставить в 01_now»
Файл остается в `01_now`, если выполняется хотя бы одно:
- Нужен для исполнения задач в текущем горизонте 1-4 недель.
- Имеет прямую связь с активным milestone в `plan.md` или execution-шагом в `tasks.md` проекта.
- Описывает оперативные решения/состояние (plan/context/tasks/log активного проекта).
- Это файл делегирования с открытыми строками (`01_now/ops/<contour>/delegations/<person-slug>.md`).
- Это `01_now/personal/tasks.md` (единственный файл личных обязательств).

## Критерии «перенести в 03_knowledge или 02_domains»
Переносить из `01_now` в стабильную базу, если выполняются 2 из 3:
- Переиспользуемость: полезно больше чем для одного проекта.
- Уникальность: содержит новые выводы/данные, не дублирует уже сохраненное.
- Объем: материал содержательнее краткой заметки-ответа.

## Критерии «архивировать в 90_archive»
- Проект завершен (`status: done`) или признан неактуальным.
- Материал больше не нужен в оперативной работе.
- Есть более свежая/каноническая версия в `03_knowledge` или активном проекте.

## Процедура закрытия проекта
1. Экстрагировать переиспользуемые знания в `03_knowledge`/`02_domains`.
2. Проставить `status: done` в `plan.md` и зафиксировать итог в `log.md`.
3. Обновить ссылки в `context.md` и README активных разделов.
4. Проверить файлы делегирований контура: открытые строки, связанные с проектом, перенести в `Archive` раздел или закрыть в трекере и отметить `Done(7d)`.
5. Перенести проектный контур в `90_archive/projects/`.
6. Удалить или пометить архивные ссылки в активных индексах (`[archived]`).

## Очистка делегирований

Отдельный цикл hygiene для `01_now/ops/<contour>/delegations/<person-slug>.md`:
- Строки в `Done(7d)` старше 7 дней — переносить в `Archive` секцию этого же файла.
- Строки в `Active` с `due:` старше 30 дней без обновления — вручную проверить в трекере (live-first-audit), либо закрыть, либо обновить.
- Раз в месяц — `forced rewrite` файлов делегирований: убрать всё, что уже не звенит, оставить только живые.
- См. [task-routing.md §delegations](./rules/task-routing.md).

## Очистка `01_now/personal/tasks.md`

- Раз в месяц — forced rewrite (удалить всё, что потеряло смысл, переформулировать оставшееся).
- Обновить `last_rewrite` в frontmatter и в теле файла.
- Не писать в него cleanup-registry записи — файл сам себя ротирует.

## Процедура очистки временных файлов
- Временные артефакты (`tmp`, `incoming`, `downloads`, тестовые выгрузки) не ссылать из стабильных заметок.
- После интеграции знаний удалять промежуточные файлы, если нет явного запроса сохранить.
- Проверять, что в `README/context` нет ссылок на временные пути.

## Ритм выполнения
- Еженедельно: короткий lifecycle-review (10-15 минут).
- Ежемесячно (1-го числа): полный `vault hygiene` по всем разделам.
- При завершении проекта: отдельная процедура закрытия (см. выше).

## Команда запуска ревью
```bash
.venv/bin/python <workspace>/scripts/vault_lifecycle_review.py \
  --root <workspace> \
  --report-md <workspace>/04_logs/weekly/$(date +%G)-W$(date +%V)_vault-lifecycle-review.md \
  --report-json <workspace>/04_logs/weekly/$(date +%G)-W$(date +%V)_vault-lifecycle-review.json
```

## Следующий шаг
После каждого запуска ревью закрывать минимум 3 пункта из отчета: 1) taxonomy/front matter, 2) переносы в `03_knowledge`, 3) архив/удаление временных файлов.
